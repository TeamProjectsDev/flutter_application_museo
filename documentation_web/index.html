Visión del Proyecto: Digitalizar el patrimonio del Museo Histórico Padre Suárez mediante una experiencia móvil híbrida que combina la realidad aumentada (AR) con una gestión de datos en tiempo real de alta fidelidad.

Narrativa y Contexto

El Museo Padre Suárez no es solo un almacén de objetos; es el alma de Granada. Esta aplicación actúa como el "Códice Digital" del museo, permitiendo que las piezas que han estado mudas durante siglos vuelvan a hablar a través de la tecnología.

Estudio de Mercado

Tras analizar aplicaciones como la del Louvre o el Prado, detectamos una falta de interactividad en museos escolares. Nuestra app llena este vacío usando AR de bajo coste pero alta precisión.

03 Arquitectura Técnica (TDD)

Arquitectura Limpia (Clean Architecture): La aplicación se divide en 4 capas independientes para asegurar que el código sea fácil de testear, mantener y escalar.

1. Capa de Interfaz (Presentation)

Es la cara visible de la app. Construida con Flutter y gestionada de forma reactiva con Riverpod. Los widgets solo se preocupan de pintar los datos y capturar los clics del usuario.

2. Capa de Negocio (Domain)

Aquí reside la "inteligencia" de la app: las reglas para subir de rango, el cálculo de recompensas y la lógica de validación de los modelos 3D.

3. Capa de Datos (Data)

Gestiona de dónde vienen los datos. Decide si debe leer la información de la Caché Local (si no hay internet) o si debe pedirla a los servidores remotos.

4. Capa Externa (Infrastructure)

Son las herramientas de terceros que usamos: Firebase para el usuario, Supabase para los archivos 3D y Stripe para los pagos.

Flujo Lógico de Escaneo (Sequence Diagram)

El proceso de escaneo no es una simple lectura; requiere validación cruzada entre Firebase (progreso) y Supabase (assets):

[Cámara] -> Captura Hash QR -> [App] Encripta Localmente -> [Supabase] Valida Existence -> [Firebase] Registra Descubrimiento -> [App] Renderiza 3D/AR

Gestión de Estados (Riverpod)

Implementamos un sistema de Estado Global Centralizado. Cada módulo (Piezas, Usuario, Pagos) tiene su propio Provider asíncrono:

// Provider para gestionar la colección del usuario en tiempo real
final userCollectionProvider = StateNotifierProvider<CollectionNotifier, CollectionState>((ref) {
  return CollectionNotifier(ref.read(repositoryProvider));
});

// Lógica de carga con manejo de estados asíncronos
final piecesFutureProvider = FutureProvider<List<Piece>>((ref) async {
  final api = ref.watch(apiServiceProvider);
  return api.fetchAvailablePieces();
});

Seguridad y Políticas RLS

Para proteger los modelos 3D y los datos de pago, usamos Row Level Security en Supabase. Solo los usuarios autenticados pueden leer los metadatos de las piezas:

-- Política de Seguridad en SQL
ALTER TABLE pieces ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Usuarios autenticados pueden ver piezas"
ON pieces FOR SELECT
TO authenticated
USING (true);

CREATE POLICY "Solo admins pueden modificar piezas"
ON pieces FOR ALL
TO authenticated
USING ( auth.jwt() ->> 'role' = 'admin' );

03.5 Especificaciones de Implementación (Deep Dive)

Ingeniería de Software: Desglose técnico de los procesos críticos: Renderizado 3D y Pasarelas de Pago.

1. Pipeline de Renderizado 3D y 360º

Para la visualización de las piezas del museo, hemos optado por el formato GLB (Binary glTF) debido a su alta compresión y soporte nativo para texturas PBR (Physically Based Rendering).

  • Librería: model_viewer_plus (basada en el motor de Google <model-viewer>).
  • Cámara 360º: Implementada mediante camera-orbit y auto-rotate. Los usuarios pueden inspeccionar cada detalle con gestos táctiles sincronizados.
  • Optimización: Aplicamos Draco Compression para reducir el tamaño de los modelos de 50MB a <5MB, permitiendo descargas fluidas.
// Fragmento de implementación del Visor 3D
ModelViewer(
  src: 'https://supabase.co/storage/v1/object/public/assets/piece_01.glb',
  alt: 'Modelo 3D Museo',
  ar: true, // Activa Realidad Aumentada nativa (ARCore/ARKit)
  autoRotate: true,
  cameraControls: true,
)

2. Arquitectura de Pagos Híbrida

Dada la naturaleza multi-plataforma de Flutter, implementamos un Payment Switcher que detecta el entorno en tiempo de ejecución para ofrecer la experiencia más nativa y segura:

A. RevenueCat (Android/iOS)

Utilizamos el SDK de RevenueCat para abstraer la complejidad de las In-App Purchases de Google Play Store.

  • Flujo: El usuario pulsa 'Donar' -> Se abre el modal nativo de Google Play -> Pago exitoso -> RevenueCat recibe el recibo firmado -> Notifica a la App mediante un CustomerInfo actualizado.
  • Seguridad: La validación del recibo se hace Server-to-Server entre Google y RevenueCat, eliminando el riesgo de 'Fake Purchases'.

B. Stripe AI Checkout (Web)

Para la versión web o donaciones directas, usamos Stripe mediante sesiones de checkout dinámicas.

  • Flujo: La app pide una CheckoutSession a Supabase Edge Functions -> Redirección a Stripe -> Pago -> Stripe envía un Webhook firmado a nuestro servidor.
  • Sincronización: No dependemos de que el usuario vuelva a la app; el servidor actualiza la tabla donations en Supabase tan pronto como recibe el evento de Stripe.
Unificación de Datos: Independientemente de la pasarela usada, ambos sistemas terminan actualizando el campo rank_level en el perfil del usuario de Firebase, asegurando que sus privilegios de "Mecenas" sean visibles en todas las plataformas.

3. Ecosistema de Datos: Firebase & Supabase

La aplicación utiliza una arquitectura de Base de Datos Híbrida para aprovechar lo mejor de cada mundo: la velocidad en tiempo real de Firebase y la potencia relacional de Supabase.

A. Firebase Cloud Firestore (El Cerebro)

Firestore gestiona los datos que cambian constantemente y requieren sincronización instantánea entre dispositivos.

  • Estructura NoSQL: Basada en colecciones (users, ranks) y documentos. Esto nos permite una escalabilidad horizontal masiva.
  • Streams de Datos: No pedimos los datos una vez; nos "suscribimos" a ellos. Si el administrador cambia la descripción de una pieza, el usuario la ve cambiar en su móvil al instante sin recargar.
// Escucha activa de datos en Flutter
FirebaseFirestore.instance
  .collection('users')
  .doc(uid)
  .snapshots() // Stream en tiempo real
  .listen((doc) {
    print("Nuevo Rango: ${doc['rank_level']}");
  });

B. Supabase (El Almacén de Alta Fidelidad)

Mientras Firebase gestiona la lógica, Supabase se encarga del almacenamiento masivo y la integridad relacional mediante PostgreSQL.

  • Supabase Storage: Aloja los modelos 3D (.glb) y las texturas de alta resolución. Usamos un sistema de Bucket Privado para que los enlaces expiren y no puedan ser compartidos fuera de la app.
  • Edge Functions (Deno): Lógica de servidor para procesar webhooks de Stripe. Esto evita que las claves secretas de pago estén dentro del código de la aplicación móvil.
Seguridad Avanzada (Row Level Security - RLS): A diferencia de otras bases de datos, Supabase permite inyectar seguridad directamente en las tablas. Hemos definido políticas SQL que validan el auth.uid() del usuario en cada petición: 
-- Ejemplo de política RLS para proteger los Modelos 3D
CREATE POLICY "Solo usuarios premium ven modelos HQ"
ON storage.objects FOR SELECT
TO authenticated
USING ( 
  bucket_id = 'models_hq' AND 
  (SELECT rank_level FROM users WHERE id = auth.uid()) >= 3
);

07 Manual de Usuario y UX

Experiencia de Usuario (UX): Diseñada para minimizar la fricción. El usuario está a solo 2 clics de la realidad aumentada desde que abre la app.

1. Registro e Inicio

El usuario puede entrar de forma anónima para probar la app. Si desea guardar su colección y subir de rango, debe vincular su cuenta con Google Auth.

2. El Escaneo Maestro

Al pulsar el botón central, se activa la cámara con un visor optimizado. Al detectar el QR de la vitrina, el móvil vibra (Haptic Feedback) y despliega la pieza 3D.

3. Interacción AR

Una vez el modelo flota en el espacio real, el usuario puede pellizcar para hacer zoom o arrastrar para rotar. Se incluye un botón de "Info" para leer la ficha técnica del Padre Suárez.

4. Donaciones

Desde la pestaña 'Tienda', el usuario puede elegir importes predefinidos (5€, 10€, 20€) o personalizados. El pago es nativo y seguro.

07.5 Modo Tester (Developer Mode)

Herramienta de QA: Sistema oculto diseñado para que los conservadores del museo y desarrolladores validen el contenido sin afectar a las métricas reales.

Acceso y Activación

Se activa mediante un "Easter Egg": pulsando 5 veces seguidas sobre el número de versión en el menú de ajustes. Esto dispara un HapticFeedback.vibrate() confirmando la activación.

El Badge de Perfil

Al activarse, el documento del usuario en Firestore se marca con is_tester: true. Esto genera un Badge flotante en su perfil que lo identifica ante el equipo técnico.
